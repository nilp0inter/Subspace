## 5. Callback Format Validation and Normalization to 16 kHz Mono PCM16

- [x] 5.1 Read the WAV file and inspect its declared format (sample rate, channel count, bit depth/encoding) from `UtteranceProgressListener.onBeginSynthesis` callback metadata and/or the written file header.
- [x] 5.2 Extend or replace the existing `WavPcmReader` (or equivalent WAV decoder helper) to expose and decode `ENCODING_PCM_8BIT`, `ENCODING_PCM_16BIT`, and `ENCODING_PCM_FLOAT` (IEEE float) from the WAV file header; decode/downmix directly to a mono normalized `FloatArray` in `[-1.0, 1.0]` (convert 8-bit unsigned by centering around zero, convert 16-bit signed by dividing by 32768, convert IEEE float by clamping to `[-1.0, 1.0]`; average left/right for stereo); do not convert to PCM16 before resampling.
- [x] 5.3 Accept mono or stereo channel count; downmix stereo to mono by averaging the two channels; reject any channel count other than 1 or 2 as a synthesis failure without attempting channel conversion.
- [x] 5.4 Resample the mono normalized `FloatArray` from the engine's native sample rate to 16 000 Hz using the existing `TtsAudio.resample(FloatArray)` linear-interpolation path, then convert the resampled f32 to PCM16 via the existing `TtsAudio.f32ToPcm16` (or equivalent) to produce `RecordedPcm(samples = ShortArray, sampleRate = 16_000)`; do not resample PCM16 directly — the resample path operates on `FloatArray`, not `ShortArray`.
- [x] 5.5 Reject any encoding other than `ENCODING_PCM_8BIT`, `ENCODING_PCM_16BIT`, or `ENCODING_PCM_FLOAT` (e.g. μ-law, A-law, ADPCM) as a synthesis failure without attempting to decode or normalize.
- [x] 5.6 Reject empty PCM output (file empty or zero samples after `onDone`) as a synthesis failure without delivering an empty `RecordedPcm` to playback.
- [x] 5.7 Reject an unparseable or malformed-header file as a synthesis failure and delete the transient file.
- [x] 5.8 On cancellation after synthesis completes but before format validation and PCM reading finish, do not deliver PCM to playback and delete the transient file.

## 7. Latest-Wins Synthesis, Recovery, and Playback Ownership
- [x] 7.1 Maintain a single `Job` reference (`activeJob`) in the navigation renderer; on each new request, follow the normal supersession order: (1) advance the generation first so it is authoritative, (2) call `TextToSpeech.stop()` to interrupt the prior utterance, (3) cancel the prior `activeJob` via `cancelAndJoin` (only when the renderer is not in a bounded recovery attempt) and wait for the prior job to fully terminate including any in-flight `synthesizeToFile` callback, transient file deletion, and `HostAudioCoordinator.play` cleanup, (4) launch the new synthesis coroutine. Do not call `stop()` after the `cancelAndJoin` — `stop()` is called before joining, not after.
- [x] 7.2 Because the generation is advanced and `stop()` is called before `cancelAndJoin`, any prior navigation synthesis callback racing the supersession is rejected by the already-authoritative new navigation generation id and the prior utterance is already interrupted; this rejection applies to navigation synthesis callbacks only, not to recovery/setup probe callbacks (6.6).
- [x] 7.3 When a second navigation announcement is requested while a prior synthesis is in progress, advance the generation first (authoritative before cancel), call `stop()` on the instance to interrupt the prior utterance, cancel-and-join the prior job, and allow only the newer request to proceed to synthesis and playback.
- [x] 7.4 When a newer navigation request arrives while a bounded runtime recovery attempt is in progress, do not cancel the recovery; replace the latest pending generation and text so the single in-flight recovery retries only the newest request (recovery coalescing: Alpha→Bravo→Journal ⇒ Journal only), and let the recovery continue to completion before synthesizing through the recovered instance.
- [x] 7.5 When a navigation announcement is requested while a PTT capture operation owns the host audio coordinator, do not preempt or interrupt the capture; respect the existing `HostAudioCoordinator` admission arbitration so the navigation playback receives `Busy` or remains pending.
- [x] 7.6 When a navigation announcement is requested while an unrelated channel-content playback is active, do not preempt the active playback; respect the existing `HostAudioCoordinator` single-owner admission.
- [x] 7.7 When a prior navigation synthesis completes and fires `onDone` concurrently with a newer request that has already advanced the navigation generation, reject the prior callback because the navigation generation is already authoritative for the new id; this applies to navigation synthesis callbacks only, not to recovery/setup probe callbacks (6.6), and allow only the newer navigation generation to be eligible for playback.
- [x] 7.8 When a navigation playback is completing its `AudioTrack` cleanup and a newer navigation request arrives, wait for `HostAudioCoordinator` to release the prior owner before admitting the newer playback, preserving the existing terminal teardown ordering.

## 9. HostAudioCoordinator.play Cancellation-Safe Teardown Ordering

- [x] 9.1 Wrap the entire post-reservation body of `HostAudioCoordinator.play` — from `strategy().acquire()` through `awaitCompletion()`, `route.release()`, and `releasePlaybackReservation` — in a single `try-finally` where all terminal cleanup runs under `NonCancellable` in the `finally` block.
- [x] 9.2 When cancellation arrives during `strategy().acquire()` (owner reserved, route not yet acquired), release no route and call `releasePlaybackReservation(operation, Interrupted)` under `NonCancellable` to clear `owner` only if `owner == operation`, clearing `activePlayback` and `rejectedPressPendingRelease`; the caller's `CancellationException` is rethrown after the `finally` cleanup completes.
- [x] 9.3 When cancellation arrives during `route.start()` (route acquired, no playback created), release the acquired route via `route.release()` then call `releasePlaybackReservation(operation, Interrupted)` under `NonCancellable` with matching-owner-only clear; the caller's `CancellationException` is rethrown after the `finally` cleanup completes.
- [x] 9.4 When cancellation arrives during `awaitCompletion()` (playback active), call `playback.skip()` to signal the independent pump to stop writing; the pump publishes `ExplicitlySkipped` completion (not `Interrupted`) because the pump itself was not cancelled — it was asked to skip via the `ActivePcmPlayback.skip()` abstraction; suspend on `awaitCompletion()` until the pump's `finally` block has called `cleanup()` (physical `AudioTrack` stop + release), call `route.release()` to release the SCO route, then call `releasePlaybackReservation(operation, Interrupted)` under `NonCancellable` with matching-owner-only clear; the caller's `CancellationException` is rethrown after the `finally` cleanup completes. `Interrupted` completion is used only when the pump coroutine itself receives `CancellationException`, not when `HostAudioCoordinator.play` is canceled and drives stop through `skip()`.
- [x] 9.5 Ensure `releasePlaybackReservation` clears `owner` only if `owner == operation` (matching-owner-only clear) in every cancellation path, not a blanket clear.
- [x] 9.6 Ensure the existing `skipOnStart` check and `close()` race paths call `playback.skip()` and `route.release()` through the same `finally` envelope so they are covered by this fix without separate handling.
- [x] 9.7 Do not release the route before stopping the track, do not clear the coordinator owner before playback completes, and do not skip `releasePlaybackReservation` on cancellation.

## 14. Bounded Runtime Recovery and Exhaustive Failure Classification

- [x] 14.1 Classify a runtime engine or service failure (`onError`, synthesis timeout, or TTS service loss) as triggering at most one bounded `TextToSpeech` reinitialization and offline-voice probe.
- [x] 14.2 On a successful reinitialization re-probe, retry only the newest pending announcement (the generation newest at the time recovery completes, which may have superseded the original trigger) and discard all older pending requests without playback; if the retry succeeds, remain `Ready` and resume navigation synthesis.
- [x] 14.3 If the re-probe reports an unusable engine, voice, or probe (engine unavailable, engine init failure, voice missing, or voice probe failure), transition bootstrap from `Ready` to `NeedsSetup` with a user-resolvable Android TTS settings or engine install action, leave `Ready`, and drop the failed generation.
- [x] 14.4 If the re-probe succeeds but the retried synthesis fails to produce valid normalized PCM, classify the failure as retryable `BootstrapState.Failed`, leave `Ready`, do not transition to `NeedsSetup`, and drop the failed generation.
- [x] 14.5 Classify an infrastructure failure (transient file I/O failure, decoder failure, PCM normalization failure, or empty PCM output after `onDone`) as retryable `BootstrapState.Failed` without any `TextToSpeech` reinitialization, leave `Ready`, and drop the failed generation.
- [x] 14.6 Distinguish same-chain exhaustion from later-independent failures: when the retried synthesis (the same recovery chain) fails for any reason — `onError`, timeout, or infrastructure — classify it as retryable `BootstrapState.Failed`, drop the generation, leave `Ready`, and do not attempt a second reinitialization in that chain; a later, fully independent engine or service failure that occurs after a fully successful recovery+retry (i.e. the prior recovery chain completed and `Ready` was restored) is a new failure chain and SHALL get one bounded recovery of its own per 14.1.
- [x] 14.7 On service teardown or bootstrap-attempt discard while the bounded reinitialization and probe are in progress, cancel the recovery attempt, shut down the reinitialized instance if the probe has not completed, clear recovery state, and invalidate the recovery probe callback by attempt identity; a newer navigation request alone SHALL NOT cancel recovery and SHALL NOT invalidate the recovery probe callback — advancing the pending navigation generation affects only navigation synthesis callback eligibility (6.6), not probe attempt callback validity.
- [x] 14.8 Under every failure classification, do not play a degraded beep or any fallback tone; the failed announcement is dropped.
- [x] 14.9 Under every failure classification, do not play partial, stale, or empty PCM; delete the transient file and drop the failed announcement or retry only per the bounded recovery rules.
- [x] 14.10 Ensure every failed path leaves `Ready` and drops the current generation; no degraded mode exists.
- [x] 14.11 Perform the recovery-to-runtime handoff atomically under a mutex: once the re-probe succeeds, acquire the mutex, publish the recovered `TextToSpeech` renderer, clear recovery state, snapshot the newest pending generation and text, and launch exactly one retry coroutine — all before releasing the mutex; this prevents a request from enqueuing between recovery completion and retry launch.
- [x] 14.12 A navigation request that arrives after the retry is enqueued supersedes the retry normally (advance generation → stop → cancelAndJoin → launch) and is not lost; the retry's generation is already stale by the time its callback fires, so its result is rejected without being counted as a failure.
- [x] 14.13 A stale cancellation of the retry (e.g. from a superseded generation) SHALL NOT be counted as a recovery-chain failure; only a genuine synthesis error, timeout, or infrastructure failure on the retried generation itself counts toward the same-chain exhaustion rule (14.6).
